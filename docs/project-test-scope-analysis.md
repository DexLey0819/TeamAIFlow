# TeamFlowAI 项目类型、技术栈与测试目标分析

## 1. 项目类型

- **Web 应用**：项目采用前后端分离架构，前端通过 Vue/Vite 提供浏览器页面，后端通过 Spring Boot 提供业务接口。
- **API 服务**：后端同时是独立 REST API 服务，接口统一以 `/api` 为主要前缀。
- **非移动应用 / 非桌面应用**：当前项目文件中未发现移动端原生框架或桌面端框架。

## 2. 技术栈和平台

### 前端技术

- Vue 3
- Vite
- Vue Router
- Pinia
- Element Plus
- Axios
- ECharts
- PlantUML 相关能力：`plantuml-encoder`

### 后端技术

- Java 17
- Spring Boot 3.3.5
- Spring MVC / REST Controller
- Spring Security
- JWT
- MyBatis-Plus
- Knife4j OpenAPI 文档
- Lombok
- Hutool

### 数据库类型

- MySQL
- 数据库脚本：`database/teamflow_ai.sql`
- 主要数据表覆盖用户、项目、项目角色、项目成员、加入申请、章节、章节内容、章节审核、评论、任务、进度、文件、AI 记录、风险快照、通知、导出记录等。

## 3. 项目功能范围

### 主要模块和功能点

- **用户认证与个人信息**
  - 注册
  - 登录
  - 当前用户信息获取
  - 个人邮箱、电话等资料编辑

- **项目管理**
  - 项目初始化 / 创建
  - 我的项目列表
  - 项目详情与概览
  - 项目 WBS 数据
  - 项目模板管理

- **项目加入与成员管理**
  - 通过项目码 / 加入链接查看项目信息
  - 提交加入申请
  - 项目经理或管理员审核加入申请
  - 项目成员列表查看
  - 项目角色与成员权限边界

- **章节与文档协作**
  - 项目章节列表
  - 章节详情
  - 章节内容保存草稿
  - 提交审核
  - 审核通过 / 退回
  - 章节评论
  - 章节版本记录
  - 章节权限矩阵
  - 文档内容中的图表、表格、PlantUML、图片等富内容渲染能力正在建设中

- **任务与进度**
  - 项目任务列表
  - 创建和更新任务
  - 任务状态流转
  - 任务日志
  - 进度记录提交
  - 项目进度状态查看

- **文件管理**
  - 项目文件上传
  - 文件列表
  - 文件下载
  - 文件删除
  - 图片文件 inline 预览接口

- **AI 辅助能力**
  - 周报生成
  - 风险分析
  - 文档检查
  - 总结报告
  - AI 记录查看
  - 从文本中提取进度信息

- **统计、导出与通知**
  - 项目完整度
  - 成员贡献
  - 风险趋势
  - 项目导出与导出历史
  - 通知列表、未读数、已读处理

- **教师与管理员功能**
  - 教师项目看板
  - 教师查看项目详情
  - 管理员项目模板管理

### 是否有复杂交互或多页面流程

是。该项目存在明显的复杂交互和多页面流程：

- 登录后访问受保护页面，并支持未登录时保留重定向地址。
- 加入项目流程跨越加入链接、登录、申请提交、项目经理审核、成员状态变更。
- 项目工作区包含概述、成员与申请、权限矩阵、章节、文件、任务、进度、AI、统计、导出等多页面。
- 章节协作包含编辑、保存草稿、提交审核、审核、退回、版本记录、评论、权限控制等状态流转。
- 文件上传与章节内容联动，包括普通附件、章节图片、inline 预览和权限边界。
- AI 生成、项目统计、导出下载等流程涉及异步请求和结果展示。

## 4. 测试目标

### 建议覆盖的测试类型

- **功能测试：需要**
  - 覆盖登录注册、项目创建、加入申请、成员审核、章节编辑审核、任务流转、进度提交、文件上传下载、AI 生成、通知、导出等核心业务流程。

- **界面测试：需要**
  - 项目是多页面 Web 应用，且大量使用 Element Plus 表单、表格、弹窗、侧边栏、上传控件、图表组件。
  - 应重点验证不同角色下菜单、按钮、表单状态、权限可见性和页面跳转。

- **接口测试：需要**
  - 后端 Controller 覆盖认证、项目、成员、加入申请、章节、权限、任务、进度、文件、AI、统计、通知、导出等 API。
  - 应重点验证 JWT 鉴权、角色权限、参数校验、状态流转和异常响应。

- **性能测试：建议**
  - 对项目列表、章节列表、文件列表、教师看板、统计图表、导出历史等列表型或聚合型接口做基础性能测试。
  - 当前项目更偏课程项目协作系统，常规性能测试优先级高于极限压力测试。

- **压力测试：可选 / 中等优先级**
  - 若后续部署到多班级、多项目并发使用场景，建议对登录、项目看板、文件上传、AI 生成、导出等高成本接口做压力测试。
  - 如果只是课程演示或小规模团队使用，压力测试可作为后续增强项。

### 是否需要自动化执行大量用例（≥50）

- **建议需要。**
- 原因：
  - 系统模块多，覆盖用户、项目、成员、章节、任务、进度、文件、AI、统计、通知、导出等多个域。
  - 存在多角色权限：学生、项目经理、教师、管理员、普通项目成员。
  - 存在多状态流转：加入申请、章节审核、任务状态、进度记录、通知已读、导出状态。
  - 存在多页面流程和接口联动，人工回归成本较高。

建议自动化用例规模：

- 接口自动化：约 40-70 条，优先覆盖核心 API、权限边界和状态流转。
- UI 自动化：约 20-40 条，优先覆盖登录、项目工作区、加入申请、章节编辑审核、文件上传、任务看板等关键路径。
- 合计自动化用例达到或超过 50 条是合理的。
