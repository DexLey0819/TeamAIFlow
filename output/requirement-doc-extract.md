### P019
1. 产品定位与目标

### P020
TeamFlowAI 不是普通的任务看板，也不是单纯的在线文档工具。它面向软件开发项目组，将项目计划、角色分工、任务协作、文档沉淀、过程审核、进度汇报、AI 分析与成果归档合并到一个统一平台中。

### P023
2. 技术栈与系统边界

### P025
系统边界：本系统重点解决软件开发项目组的协作管理和文档沉淀问题，不做即时聊天、在线视频会议、Git 代码托管、自动生成完整代码和企业级复杂流程引擎。

### P037
7. 角色权限与板块模型

### P038
系统采用“项目模板 -> 项目角色 -> 项目板块 -> 板块权限”的模型。项目创建后根据开发模式生成默认角色、必要板块和权限规则。项目性质创建后不可修改，但项目经理可在权限允许范围内增加扩展板块。

### P041
8. AI 智能体能力设计

### P042
本项目建议使用多个专职智能体，而不是一个通用 AI 按钮。每个智能体拥有不同数据输入、Prompt 模板、输出结构和保存位置。

### P044
AI 输出必须带有“基于已提供数据分析，不编造未出现内容”的约束。系统可以在 AI 调用失败时提供本地演示生成逻辑，保证课程演示稳定。

10. 用例设计（80个）

### P048
本节提供 80 个用例，覆盖项目创建、成员加入、角色权限、文档板块、任务、进度、AI、统计、通知、模板、导出和教师查看等核心场景。用例不以简单增删改查为主，而强调业务流程、权限约束、异常规则和协作链路。

### P053

### Table 1 (1x1)

### Table 2 (6x2)
目标 | 说明
规范项目启动 | 通过项目开发模式、角色人数上限、固定板块和权限模板，让项目从创建时就具备明确结构。
强化项目协作 | 通过任务、板块、批注、审核、通知、进度汇报形成可追踪的协作过程。
沉淀项目成果 | 把需求、用例、页面、数据库、接口、测试、总结等成果转化为结构化项目文档。
引入 AI 辅助管理 | 使用不同 AI 智能体完成周报生成、风险识别、文档缺失检查和项目总结。
支持课程提交和项目归档 | 将项目过程和最终成果导出为 PDF，便于课程验收、答辩和复盘。

### Table 3 (9x2)
类别 | 技术 / 说明
系统架构 | 前后端分离架构，前端通过 RESTful API 调用后端服务。
前端技术 | Vue 3、Vite、Element Plus、Vue Router、Pinia、Axios、ECharts。
后端技术 | Java 17、Spring Boot 3.3.5、Spring Web、Spring Security、Spring Validation、MyBatis-Plus、Lombok、Hutool。
数据库 | MySQL 8；核心表包括用户、项目、项目成员、项目角色、项目板块、任务、任务日志、进度记录、项目文件、AI 生成记录、通知等。
权限认证 | JWT、Spring Security、BCrypt 密码加密、无状态会话认证。
AI 能力 | 支持 DeepSeek 等兼容 Chat Completions 格式的大模型 API，也提供本地演示生成逻辑。
接口文档 | Knife4j / OpenAPI 3。
运行方式 | Maven 管理后端依赖，npm 管理前端依赖；后端 8080，前端 5173，Vite 代理 /api 到后端。
